# 02 - youki Integration Plan

## Target

Implement a real OCI runtime-spec execution path by invoking `youki` inside Linux guest VMs, while keeping host-side OCI image management in `vz-oci`.

## Current State Snapshot

- `Runtime::run` in `crates/vz-oci/src/runtime.rs` pulls image + assembles rootfs + boots VM.
- `ExecutionMode::OciRuntime` is currently a placeholder and uses guest exec path.
- No OCI bundle lifecycle (`config.json` + runtime state dir) exists in runtime path today.

## Linux Guest Agent Identity (Authoritative)

Linux OCI lifecycle handling belongs to the Linux guest agent path, not the macOS guest-agent crate.

- Linux runtime agent code path: `crates/vz-linux/src/agent.rs` (and Linux initramfs bootstrap).
- macOS guest-agent crate: `crates/vz-guest-agent` remains macOS-focused and does not own Linux OCI dispatch.

All OCI dispatch handlers defined in this document are implemented in the Linux agent path.

## youki Provisioning Strategy

`B04` in `04-beads.md` establishes runtime binary provisioning.

v1 delivery path:

- Build/download pinned `youki` for `linux/arm64`.
- Package it in Linux guest artifacts alongside kernel/initramfs metadata.
- Expose it to guest via a dedicated VirtioFS share on boot.
- Install or bind-mount to `/usr/local/bin/youki` in guest bootstrap.
- Validate version and checksum before use.

v2 hardening path:

- Bake `youki` into initramfs/guest artifact image for fully hermetic boot.

## Integration Model

Host responsibilities:

- Pull image and assemble rootfs.
- Generate OCI `config.json` from image config + run overrides.
- Prepare shared directories for bundle + mounts + runtime binary delivery.
- Drive VM lifecycle and reconcile service state.

Guest responsibilities:

- Provide callable `youki` binary and runtime dependencies.
- Materialize runtime root (`/run/vz-oci/...`) and persistent container state dir.
- Execute runtime lifecycle commands and return structured results.

## Bundle and Filesystem Layout

Inside guest:

```text
/var/lib/vz-oci/
  bundles/<container-id>/
    config.json
    rootfs/         # overlay-mounted working root
  state/<container-id>/
    meta.json
```

Host shared dirs into VM (VirtioFS):

- `rootfs-ro-<id>`: assembled read-only rootfs tree.
- `bundle-<id>`: bundle content generated by host.
- `mount-*`: user bind/named volume paths resolved by mount planner.
- `oci-runtime-bin`: runtime binary delivery share (`youki`).

## Overlay and Mount Setup

Guest mount model for each container:

- Lower: read-only host rootfs (`rootfs-ro-<id>`).
- Upper/work: writable guest-local dirs under `/var/lib/vz-oci/overlay/<id>/`.
- Merge mount becomes bundle `rootfs/`.

Mount ordering:

1. Create overlay merged root.
2. Apply bind/named mounts from bundle config in deterministic order.
3. Start runtime lifecycle.

This is implemented by `B10` in `04-beads.md`.

## Lifecycle Mapping

`vz-oci` API to guest `youki` command mapping:

- `create` -> `youki create <id> --bundle <path>`
- `start` -> `youki start <id>`
- `state` -> `youki state <id>`
- `exec` -> `youki exec <id> -- <cmd...>`
- `stop` -> `youki kill <id> SIGTERM` then `SIGKILL` on timeout
- `remove` -> `youki delete <id>`

## Protocol Changes

Add typed agent requests to avoid shell-string coupling:

- `OciCreate { id, bundle_path }`
- `OciStart { id }`
- `OciState { id }`
- `OciExec { id, command, args, env, cwd, user }`
- `OciKill { id, signal }`
- `OciDelete { id, force }`

Responses:

- `OciOk { id, payload }`
- `OciError { id, code, message }`

## RuntimeConfig and RunConfig Extensions

`RuntimeConfig` additions:

- `guest_oci_runtime: OciRuntimeKind` (default `Youki`)
- `guest_oci_runtime_path: Option<PathBuf>`
- `guest_state_dir: Option<PathBuf>`

`RunConfig` additions:

- `mounts: Vec<MountSpec>`
- `init_process: Option<Vec<String>>` (entrypoint override semantics)
- `oci_annotations: Vec<(String, String)>`

Dependency decision:

- Use the `oci-spec` crate for runtime-spec `config.json` modeling/serialization.
- Do not hand-roll runtime-spec structs.

## Integration Waves

Wave 1 (minimum viable compliant path):

- Keep existing guest-agent transport.
- Add runtime binary provisioning.
- Add typed OCI protocol + Linux agent dispatch.
- Execute lifecycle parity for create/start/state/kill/delete.

Wave 2 (operational parity):

- Add strict OCI config generation module.
- Add mount planner + overlay setup and compose-compatible volume semantics.
- Add stable runtime state persistence and crash recovery re-sync.

Wave 3 (post-v1 enhancements):

- Checkpoint/restore support if kernel/runtime constraints permit.
- Broader OCI hook coverage where available.

Wave 3 is explicitly out of scope for the current bead plan and tracked as follow-on work.

## Failure and Recovery Model

Rules:

- Runtime state source of truth is `youki state`.
- Host state store caches observed state but must re-sync on restart.
- Container teardown must be idempotent (safe to re-run delete/cleanup).
- VM crash recovery is validated before weekly stress gate.

## Security Notes

- Do not pass untrusted command strings to shell inside guest for lifecycle ops.
- Keep runtime requests typed and argument-separated.
- Validate all host mount inputs before VM creation.
- Enforce runtime binary checksum/version verification before execution.
